# synthwatch (runner) — Claude rules

Rules Claude should follow when working in this repo. The runner is a TypeScript/Node ACA cron job that OWNS all Postgres schema + migrations; synthwatch-api (.NET) and synthwatch-dashboard (Next.js) are read-mostly consumers.

## Lessons from 2026-07-02

- **CONCURRENTLY index migrations carry NO `BEGIN/COMMIT`.** `db/migrate.sh` applies each file with a bare `psql -f` (autocommit), which is exactly what `CREATE INDEX CONCURRENTLY` requires (it can't run in a transaction). Use `IF NOT EXISTS` + role-guard; template is `db/migrations/0032_incidents_opened_idx.sql`. schema.sql gets the same index non-concurrently for fresh installs. *(from #181 — the stale-'running' reap partial index)*
- **A new ACA cron job must be wired in THREE places or CI fails:** the `RUNNER_IMAGE_JOBS` array in `scripts/lib/deploy-lib.sh` (single source of truth), the `az containerapp job update -n …` list in `.github/workflows/deploy.yml`, and the bicep resource in `infra/main.bicep`. The #180 parity guard (`scripts/deploy_test.sh`) asserts deploy.yml rolls EXACTLY `RUNNER_IMAGE_JOBS`, so a one-sided edit fails ci-gate. *(from #180, #182)*
- **The runner owns ALL Postgres schema + grants, but the `checks` table is bootstrapped from `db/schema.sql` — no migration `CREATE`s it.** So a constraint/grant can be present in prod without any migration adding it (it came from schema.sql or out-of-band ops), and a fresh env replayed from migrations alone would NOT reproduce it. Before assuming a "missing migration," confirm the object's provenance: `grep` migrations, then check schema.sql, then check prod with `\d`/`pg_constraint`. *(from the browser_needs_flow diagnostic — the review claimed prod lacked a constraint that prod actually had)*
- **`incidents.opened_run_id`/`resolved_run_id → runs(id)` is RESTRICT (no `ON DELETE` clause), NOT cascade.** A naive `DELETE FROM runs …` is blocked by any incident-pinned run. Row-retention must exclude incident-pinned runs (`NOT EXISTS (… incidents …)`). `run_steps`/`run_metrics` ARE `ON DELETE CASCADE` and clean up with the run. *(from #182 — row-retention job)*
- **The row-retention window is tied to `infra/main.bicep artifactRetentionDays` (the 90d blob lifecycle)** — rows and their trace/screenshot blobs must expire on the same clock or you get dangling `trace_url`/`screenshot_url`. `RETENTION_DAYS` carries a header contract + a unit assertion pinning it to 90. *(from #182)*
- **API-role Postgres grants are split-provenance and the CI gate is partial.** SELECT-for-all-tables is out-of-band ops (`ALTER DEFAULT PRIVILEGES`, in neither migrations nor schema.sql); `checks` writes are out-of-band too. The `synthwatch-api/scripts/check-pg-grant-coverage.mjs` gate is writes-only + a pure static parse — it never creates the role or asserts real grants, so a missing SELECT grant is invisible until a prod 500. Don't trust "the grant gate is green" as proof a grant exists. *(from the grant-baseline diagnostic)*
- **DB integration tests are gated on `SKIP = !process.env.DATABASE_URL` and run in CI via the Postgres-service job (#173).** But CI + the api's Testcontainers run as SUPERUSER with no `synthwatch-api` role, so a missing GRANT is invisible to every test — it only surfaces as a prod 500 (cf. missing DELETE #0044, missing UPDATE #0053). *(from #173 + grant diagnostic)*
- **Runner↔C# extractor parity is a recurring divergence class — guard it with SHARED golden fixtures.** When changing a trace-signal/URL extractor, update the shared fixture rather than letting the two implementations drift (#170), and store network/mutation URLs RAW (not redacted) to keep byte-parity with the C# extractor (#171 then #172, which completed what #171 only half-fixed). *(from #169–#172)*
- **When characterizing the `deploys` table (or any append-heavy table), aggregate — don't read a recent-N view.** A `SELECT * … ORDER BY deployed_at DESC LIMIT 10` is dominated by the frequently-redeploying dashboard's etag rows and hides older `sentry-release`/`is_sha=t` rows; `GROUP BY target_host, source, is_sha` tells the truth. *(from the deploy-marker diagnostic — a "prod is all etag" claim was a recent-N artifact)*

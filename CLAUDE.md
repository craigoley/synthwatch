# synthwatch (runner) ‚Äî Claude rules

Rules Claude should follow when working in this repo. The runner is a TypeScript/Node ACA cron job that OWNS all Postgres schema + migrations; synthwatch-api (.NET) and synthwatch-dashboard (Next.js) are read-mostly consumers.

## Lessons from 2026-07-02

- **CONCURRENTLY index migrations carry NO `BEGIN/COMMIT`.** `db/migrate.sh` applies each file with a bare `psql -f` (autocommit), which is exactly what `CREATE INDEX CONCURRENTLY` requires (it can't run in a transaction). Use `IF NOT EXISTS` + role-guard; template is `db/migrations/0032_incidents_opened_idx.sql`. schema.sql gets the same index non-concurrently for fresh installs. *(from #181 ‚Äî the stale-'running' reap partial index)*
- **A new ACA cron job must be wired in THREE places or CI fails:** the `RUNNER_IMAGE_JOBS` array in `scripts/lib/deploy-lib.sh` (single source of truth), the `az containerapp job update -n ‚Ä¶` list in `.github/workflows/deploy.yml`, and the bicep resource in `infra/main.bicep`. The #180 parity guard (`scripts/deploy_test.sh`) asserts deploy.yml rolls EXACTLY `RUNNER_IMAGE_JOBS`, so a one-sided edit fails ci-gate. *(from #180, #182)*
- **The runner owns ALL Postgres schema + grants, but the `checks` table is bootstrapped from `db/schema.sql` ‚Äî no migration `CREATE`s it.** So a constraint/grant can be present in prod without any migration adding it (it came from schema.sql or out-of-band ops), and a fresh env replayed from migrations alone would NOT reproduce it. Before assuming a "missing migration," confirm the object's provenance: `grep` migrations, then check schema.sql, then check prod with `\d`/`pg_constraint`. *(from the browser_needs_flow diagnostic ‚Äî the review claimed prod lacked a constraint that prod actually had)*
- **`incidents.opened_run_id`/`resolved_run_id ‚Üí runs(id)` is RESTRICT (no `ON DELETE` clause), NOT cascade.** A naive `DELETE FROM runs ‚Ä¶` is blocked by any incident-pinned run. Row-retention must exclude incident-pinned runs (`NOT EXISTS (‚Ä¶ incidents ‚Ä¶)`). `run_steps`/`run_metrics` ARE `ON DELETE CASCADE` and clean up with the run. *(from #182 ‚Äî row-retention job)*
- **The row-retention window is tied to `infra/main.bicep artifactRetentionDays` (the 90d blob lifecycle)** ‚Äî rows and their trace/screenshot blobs must expire on the same clock or you get dangling `trace_url`/`screenshot_url`. `RETENTION_DAYS` carries a header contract + a unit assertion pinning it to 90. *(from #182)*
- **API-role Postgres grants are split-provenance and the CI gate is partial.** SELECT-for-all-tables is out-of-band ops (`ALTER DEFAULT PRIVILEGES`, in neither migrations nor schema.sql); `checks` writes are out-of-band too. The `synthwatch-api/scripts/check-pg-grant-coverage.mjs` gate is writes-only + a pure static parse ‚Äî it never creates the role or asserts real grants, so a missing SELECT grant is invisible until a prod 500. Don't trust "the grant gate is green" as proof a grant exists. *(from the grant-baseline diagnostic)*
- **DB integration tests are gated on `SKIP = !process.env.DATABASE_URL` and run in CI via the Postgres-service job (#173).** But CI + the api's Testcontainers run as SUPERUSER with no `synthwatch-api` role, so a missing GRANT is invisible to every test ‚Äî it only surfaces as a prod 500 (cf. missing DELETE #0044, missing UPDATE #0053). *(from #173 + grant diagnostic)*
- **Runner‚ÜîC# extractor parity is a recurring divergence class ‚Äî guard it with SHARED golden fixtures.** When changing a trace-signal/URL extractor, update the shared fixture rather than letting the two implementations drift (#170), and store network/mutation URLs RAW (not redacted) to keep byte-parity with the C# extractor (#171 then #172, which completed what #171 only half-fixed). *(from #169‚Äì#172)*
- **A migration touching a SHARED table (one the api maps ‚Äî its `DbContext` / `tests/**/fixtures/schema.sql` is the list) REDS synthwatch-api's required schema-parity gate and freezes api merges until that fixture is patched to match.** Deliberate drift-forcing (catches the incident-2 class that shipped a prod 500), not a bug: plan the api fixture patch in the same or an immediately-following synthwatch-api PR, and diagnose a frozen api queue as "I touched a shared table." See `db/migrations/README.md`. *(from the synthwatch-api #167 parity gate's accepted advisory)*
- **When characterizing the `deploys` table (or any append-heavy table), aggregate ‚Äî don't read a recent-N view.** A `SELECT * ‚Ä¶ ORDER BY deployed_at DESC LIMIT 10` is dominated by the frequently-redeploying dashboard's etag rows and hides older `sentry-release`/`is_sha=t` rows; `GROUP BY target_host, source, is_sha` tells the truth. *(from the deploy-marker diagnostic ‚Äî a "prod is all etag" claim was a recent-N artifact)*

## Lessons from 2026-07-12

- **`printf "$var" | grep -q` (or `grep -qxF`) under `set -o pipefail` is a SIGPIPE trap that yields a VACUOUS/wrong result.** grep -q short-circuits on the FIRST match and closes the pipe ‚Üí printf takes SIGPIPE (exit 141) ‚Üí under pipefail the *pipeline's* status is 141, not grep's 0 ‚Üí an `if ! ‚Ä¶` inverts to the wrong branch (plus a `printf: write error: Broken pipe`). This bit `scripts/deploy.sh` THREE times: #155 (SHA tag membership), #279 (CORS verify printed "template declares no blob CORS" while CORS *was* declared + live), #283 (the SHA_OVERRIDE sibling). Use a here-string (`grep -qxF "$x" <<<"$list"`) or pure-bash `[[ "$var" == *needle* ]]` ‚Äî no dangling writer to SIGPIPE. *(from #279, #283 ‚Äî the #155 fix left two siblings unswept)*
- **A `verify()`/gate check must FAIL (not vacuously PASS) when it cannot actually compare.** `verify()` is the platform's primary drift guard, so a check that reports green while asserting nothing manufactures confidence ‚Äî worse than no check. Guard every comparator against an empty expected side (`num_eq`/`mem_eq`/`str_eq` all `return 1` on empty-$1), flunk on empty-live and empty-vs-empty, and print a distinct `SKIP` (never `PASS`) for a legitimately-inapplicable check. Add a per-comparator must-go-red (empty-expected / empty-live / both-empty all FLUNK). *(from #279, #281 ‚Äî the vacuous-CORS sweep)*
- **`CREATE OR REPLACE FUNCTION` cannot change a function's RETURN TYPE ‚Äî Postgres requires a `DROP FUNCTION` first.** And both a fresh DB and synthwatch-api's `check-schema-parity.sh` materialize the schema as `db/schema.sql` (the end-state) THEN replay every migration idempotently on top. So a migration that redefines a function's signature MUST be `DROP FUNCTION IF EXISTS f(<exact old argtypes>); CREATE FUNCTION f(...)` ‚Äî a `CREATE OR REPLACE` errors `cannot change return type of existing function` at that migration once schema.sql carries the new signature, halting the replay (breaking fresh DBs / restores / the parity gate). When you update a function's return type in schema.sql, re-check that OLD migrations still replay on top. *(from #284 ‚Äî 0069 cost_projection stayed CREATE OR REPLACE after #280 changed its return type + schema.sql)*
- **Amending an ALREADY-APPLIED migration is safe IFF the edit is EFFECT-PRESERVING** ‚Äî body byte-identical, only adding replay-idempotency (e.g. `DROP ‚Ä¶ IF EXISTS` before the `CREATE`). Prod recorded the migration in `schema_migrations`, so `db/migrate.sh` NEVER re-runs it; the amendment changes nothing in any DB that already applied it and only fixes the from-scratch replay. Prove it with a REAL local Postgres replay (`schema.sql` + all migrations): the amended version converges AND the original fails at that migration. *(from #284 ‚Äî chose "edit 0069 to DROP+CREATE" over a new migration, which can't fix a failure that halts at 0069)*
- **A runner tick runs its due checks SEQUENTIALLY in ONE ACA job execution, sharing that execution's 660s `replicaTimeout`** (`runner/index.ts`: `drainRunRequests()` awaited, then `for (const c of due) await runOne(c)`). So a task needing a *fresh* time budget (e.g. a confirmation re-run of a long browser flow) must fire a DEDICATED off-cadence execution via ARM `jobs/start`, NOT rely on the next `*/5` cron tick (which shares the sequential budget). The reusable off-schedule machinery is `run_requests` ‚Üí `drainRunRequests` ‚Üí `forceClaim` (no due-predicate, bypasses the interval) ‚Üí `runOne`; `sandbox`/`confirmation` are per-request "flavor" flags mirrored the same way. *(from #278 ‚Äî confirmation-retry as a fresh execution, 0077)*
- **DNS checks: "resolves" ‚âÝ "resolves correctly."** A `dns` check with no `net_config.expectedValue` stays GREEN on a wrong-but-live repoint (hijack / botched CDN cutover / stale-but-answering failover). `runner/netChecks.ts` supports `recordType: CNAME` + a **substring** `expectedValue` ‚Äî for CDN-fronted hosts assert the STABLE CNAME target (e.g. `wegmans.com.edgekey.net`), NOT an A record (edge IPs rotate ‚Üí constant false reds). An APEX has no CNAME ‚Üí document a deliberate **liveness-only** gap, don't A-pin (it flaps on a vendor IP migration). ‚òÖ False-red tell: a DNS check red while the BROWSER check for the same host is GREEN = a benign CDN migration; both red = a real repoint. *(from #282 + the DNS 346‚Äì349 recon)*

## Lessons from 2026-07-21

- **`compileSpec` supports exactly ONE import form, and `@playwright/test` is ALIASED to the shim ‚Äî not made external.** Every spec in synthwatch-monitors imports `'../../lib/flow'`, which `LIB_FLOW_RE` rewrites to the shim placeholder, so that was the only form either path had ever compiled. The Tests area, though, receives specs an operator PASTED, and those open with `import { test, expect, type Page } from '@playwright/test'` ‚Äî a name nothing handled (`external: ['playwright']` is a different package, and `resolveDir` is the OS temp dir with no `node_modules`), so the preview died at `Could not resolve "@playwright/test"` and the Tests area was never usable for its purpose. ‚òÖ The fix routes it to the **same** shim placeholder as lib/flow. Marking it `external` would also have silenced the compile error ‚Äî and produced a child loading the REAL Playwright runner: no step recording, no captured-test registry, semantics unlike production. **A preview that compiles differently than production is a preview that lies.** ‚òÖ CAVEAT: the shim implements `SUPPORTED_MATCHERS`, not all of Playwright's, so a pasted spec using an exotic matcher COMPILES and then fails at RUNTIME with the shim's own message ‚Äî that is the honest failure mode, and better than silently running uninstrumented. ‚òÖ Note prod was never affected: the one monitor mentioning the package does so in a TYPE position (`import('@playwright/test').Request`, meals2go-cheese-pizza-cart.spec.ts:255), which the ts loader erases. ‚òÖ There is only ONE compile config ‚Äî `runSandboxPreview.ts` calls the same `compileSpec` as the real check path; do not go looking for a sandbox fork to unify. *(from #349 ‚Äî sandbox execution 27m12hm; the earlier "successes" were trivial lib/flow specs, decoded from ACA execution history)*

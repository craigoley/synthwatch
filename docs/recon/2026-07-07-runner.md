**Falsifier (run):** "if a diff/canonicalization function exists in the runner, grep for `Diff`,

### What I *can* ground-truth: the payload the diff consumes is target-host-STAMPED

The diff (wherever it runs) compares two runs' `trace_signals` JSON. That payload is **not**
env/target-neutral — it is stamped with the target host in three places (`runner/traceSignals.ts`):

- `targetHost: string | null` is a top-level field of `TraceSignals` (`:59`), set from the run's
  `targetUrl` (`:107` `hostOf(targetUrl ?? '')`).
- Every network request stores its URL **RAW** — `slim()` copies `url: r.url` (`:174-183`) and the
  header comment (`:169-172`) is explicit: "Store the url RAW … Network urls are NEVER redacted."
  Mutations likewise store raw URLs (`:229-232`).
- `thirdParty` is computed **relative to the target host** (`:164` `third: !isSite(hostOf(url), targetHost)`),
  and `isSite` (`:312-317`) is host-suffix matching. Console `origin` is `'site'` vs `'third-party'`
  by the same target-relative test (`:270`).

**INFERRED (from the payload, not the diff):** a raw diff of run-A (staging host) vs run-B (prod
host) `trace_signals` would be dominated by host/URL differences — every `url` differs, `targetHost`
differs, and `thirdParty`/`origin` classifications can flip when the target host changes. So
env-agnostic diffing is **only** possible if the diff layer canonicalizes host/URL noise away
(analogous to the WebSocket-id/timestamp stripping the question describes). Whether it does that is a
property of the out-of-repo API diff, **not verifiable here**.

### VERDICT

Cannot be answered from the runner repo — the diff logic named in the question lives in
`synthwatch-api`. **To answer whether a staging run can be diffed against a prod run of the same
monitor with only a caller change, read `synthwatch-api`'s trace-diff implementation** (the
`GetTraceSignals` consumer): check whether it normalizes `targetHost` / request URLs / the
`thirdParty` flag before comparing. From the runner side the signal is a **caution**: the payload is
host-stamped, so a naive pair-keyed diff is *not* environment-agnostic without canonicalization in
the diff layer. **Caveat:** this repo cannot confirm the diff exists or how it keys; do not treat
"the runner ports the extractor" as evidence about the diff's env-handling.

## 2. ★S3 — Trace DOM/screenshot sufficiency for selector-heal

### ANSWER (GROUND TRUTH): a non-sensitive failed run retains enough to propose a heal; a sensitive one does not.

**What is captured on a browser FAILURE** (`runner/index.ts`):

- Tracing is started for the whole run with `context.tracing.start({ screenshots: true, snapshots: true, sources: false })`
  (`:930-931`).
- On the failure branch (catch): `screenshot = await page.screenshot()` (`:1004`).
- In the `finally`, when `failed`, the trace is written to a zip: `context.tracing.stop({ path: tracePath })`
  (`:1013-1015`).
- `page.content()` (the full DOM HTML) is captured at `:995` — but **only in the PASS branch**, and
  **only** as a transient input to `noteDeployMarker` (deploy-marker detection). It is **not**
  persisted and **not** captured on failure. So there is no standalone stored DOM-HTML column.

**Playwright primary-doc verification (OBSERVED):** `tracing.start` docs
(https://playwright.dev/docs/api/class-tracing) — `snapshots`: "If this option is true tracing will
capture **DOM snapshot on every action** [and] record network activity"; `screenshots`: "capture
screenshots … used to build a timeline preview." So the persisted **trace zip contains a DOM snapshot
per action** plus per-action screenshots. This is the DOM source for a heal — **not** `trace_signals`.

**What is persisted per run** (`runs` UPDATE, `runner/index.ts:634-660`): `error_message`,
`failed_step`, `screenshot_url`, `trace_url`, `trace_signals` (`::jsonb`). Plus per-step rows in
`run_steps` (`stepRecorder.ts:66-79`): `name`, `status`, `duration_ms`, `error_message`.

**The failed selector is retained in text.** `StepRecorder.step` catch sets
`this.failedStep = name` (`stepRecorder.ts:133`) and persists the Playwright error message
(`:124,:131-132`) — e.g. `TimeoutError: locator '.cuisine-tile' not found`. `scrubError`
(`redact.ts:88-100`) is explicit that for a sensitive monitor it keeps `"TimeoutError: locator
'.cuisine-tile' not found"` readable while removing secrets. So `runs.error_message` +
`runs.failed_step` + `run_steps.error_message` carry **the failed selector and the step it died in.**

**What `trace_signals` does NOT contain:** the `TraceSignals` interface (`traceSignals.ts:58-62`) is
`{ targetHost, network, console }` — **no DOM, no accessibility tree, no selector, no element
geometry.** The extraction only reads the `trace.network` and `trace.trace` NDJSON entries
(`:110-111`), never the DOM-snapshot resources. So the canonicalized `trace_signals` JSON is
insufficient for a heal on its own — but it is a **separate artifact** from the trace zip; it does
**not** strip DOM *from the zip*.

**What redaction / the persist policy STRIPS** (`redact.ts`):

- `makeRedactor` / the built-in denylist scrub **network-URL and console TEXT** only (`redact.ts:2,16-25`),
  and only for **sensitive** monitors (`IDENTITY_REDACTOR` = byte-identical for non-sensitive, `:29-30`).
  Redaction never touches a DOM snapshot.
- `tracePersistPlan` (`redact.ts:65-72`) is the load-bearing gate:
  - **Non-sensitive, status fail/error:** `failureTrace: true`, `failureScreenshot: true` → the DOM-bearing
    trace zip **and** the screenshot are stored (`index.ts:567-568, 594-596`).
  - **Sensitive (any status):** `{ failureTrace:false, successBaseline:false, failureScreenshot:false, baselineScreenshot:false }`
    → **no zip, no screenshot, no baseline** (`index.ts:590-593` comment: "a sensitive monitor stores
    NO raw trace zip … carry the full session-bearing DOM/network").

**Falsifier (run):** "if a failed non-sensitive run kept no DOM, tracing would be started without
`snapshots`, or the zip wouldn't be saved on failure." → Refuted: `snapshots:true` at `index.ts:931`
+ `tracing.stop({path})` under `if (failed …)` at `:1013-1015` + `failureTrace:true` at `redact.ts:71`.
**Second falsifier:** "if sensitive monitors kept a healable artifact" → Refuted: `redact.ts:67`
returns all-false.

### VERDICT

- **Non-sensitive monitor:** **YES.** A failed run retains (a) the failed selector + failing step name
  (`runs.error_message` / `failed_step` / `run_steps`), (b) the failure screenshot (`screenshot_url`),
  and (c) **per-action DOM snapshots inside the trace zip** (`trace_url`, `snapshots:true`). An AI
  that opens the trace zip has the DOM at the failing step + the moved-from selector + a screenshot —
  enough to propose where a moved element went. **Caveats:** the heal path must read the **trace zip's
  DOM snapshots**, not `trace_signals` (which has no DOM); and the failure zip rides the **90d artifact
  purge** (`index.ts:572`) — heal must run within 90d. No explicit **accessibility tree** is captured
  (Playwright snapshots are DOM, not the a11y tree).
- **Sensitive monitor:** **NO.** `tracePersistPlan` suppresses the zip, the screenshot, and the
  baseline; only a **scrubbed** `error_message` (+ `failed_step` + redacted `trace_signals`) survives.
  A heal cannot locate a moved element. This is by design (session-bearing DOM/PII), an accepted
  trade.
- **Does canonicalization/redaction remove DOM a heal needs?** For non-sensitive: **no** — signal
  extraction is a separate summary and never edits the zip. For sensitive: **yes, effectively** — the
  whole DOM-bearing zip is never stored.

## 4. HYPOTHESIS — why a full bicep apply didn't place `SYNTHWATCH_DEPLOYED` on the running jobs (T6.6)

**Research question, not a fix. Conclusion is a HYPOTHESIS; no apply attempted.**

### Repo facts (OBSERVED)

- The recovery deploy path is `az deployment group create` (`origin/main:scripts/deploy.sh:471`), **no
  `--mode`** → Incremental (default). It is **not** a deployment stack (`az stack …` appears nowhere;
  only `az deployment group what-if` / `create`).
- The marker is **unconditionally** in the job template: `infra/main.bicep` primary job
  (`Microsoft.App/jobs@2024-03-01`) has `properties.template.containers[0].env` with
  `{ name:'SYNTHWATCH_DEPLOYED', value:'1' }` (bicep ~`:99-100`), and the same on all 8 jobs (§ prior
  incident). No `condition:` on the resource, no parameter gating the env entry.

### Candidate mechanisms, each checked against Azure primary docs

**(1) Deployment-stack `excludedActions` / `*/read` — REFUTED for this path.** `excludedActions` is a
[deployment-stacks](https://learn.microsoft.com/azure/azure-resource-manager/bicep/deployment-stacks)
feature; deploy.sh uses plain `az deployment group create`, so no stack denyAssignment/excludedActions
is in the loop. *Residual:* if a stack managed the RG out-of-band it couldn't be seen without `az` —
low prior, listed for completeness.

**(2) Env merge-vs-replace on the job via ARM — the doc says REPLACE, which makes absence surprising.**
ARM deployment-modes doc (primary): "When redeploying an existing resource in incremental mode, **all
properties are reapplied. The properties aren't incrementally added.** … The resource definition in the
template always contains the final state of the resource." So a successful PUT of the job with the
marker in `env` would **set** the marker (and would even *overwrite* a marker-less live env). Merge is
not the mechanism — a completed PUT should have fixed it. This sharpens the mystery: the PUT's *effect*
was absent, so either the PUT didn't run or didn't carry the marker.

**(3) Revision-scope vs job-scope env — REFUTED.** Container Apps **jobs have no revisions**
(primary jobs doc describes only *Job* → *Job execution* → *Job replica*; revisions are an Apps
concept). "A job defines the default configuration that's used for each job execution." So there is no
stale revision pinning old env — a control-plane template change applies to the **next execution**.
Therefore, had the control-plane template carried the marker after the apply, subsequent executions
would have had it. That they didn't means the **control-plane job template still lacked the marker
after the apply**.

**(4) what-if vs apply divergence — plausible but not self-sufficient.** what-if predicts; apply
writes. Divergence alone doesn't leave env unset unless the apply also no-op'd or partially failed.

### BEST-SUPPORTED HYPOTHESIS

**ARM's incremental idempotency took *no operation* on the `Microsoft.App/jobs` resources — the
declarative deploy never issued the env-bearing PUT — so the running jobs' template stayed
marker-less, while the imperative `az containerapp job update --set-env-vars` wrote the env directly
and thus was the only thing that worked.**

This rests on the *exact* documented Incremental behavior (primary): **"If the resource already exists
in the resource group and its settings are unchanged, no operation is taken for that resource."** The
mechanism is: the automated CD mutates each job imperatively (`az containerapp job update --image`,
`deploy.yml`), so by recovery time the **live control-plane resource had drifted from what
`az deployment group create` computed as its diff baseline**; if ARM's change-detection for the job
concluded "unchanged," it skipped the PUT entirely — the env array was never rewritten. `--set-env-vars`
does a targeted PATCH of the env collection and cannot be idempotency-skipped, which is why it, and only
it, placed the marker. (Consistent with #202's finding that VERIFY was *blind* to the marker: a
skipped-PUT deploy would still report "Succeeded," and the old VERIFY never looked at the env, so
"apply succeeded + fleet still down" was invisible.)

Secondary (kept, lower prior): a **partial Incremental deployment** — an earlier resource in the
template errored and ARM stopped before the job env PUT (the documented "Failed-but-landed"
inverse from `scripts/deploy.sh:10-11`). Weaker because the deploy reportedly reported success.

Explicitly checked and **not applicable**: the deployment-modes **child-resource dual-representation
gotcha** ("site configuration values for web apps … specify an empty object → child isn't updated")
— that concerns `Microsoft.Web/sites/config` and VNet subnets, not `Microsoft.App/jobs` env, which is
an inline array on the job's own `template`.

### What would CONFIRM it (Craig runs; do NOT apply)

1. **`az deployment operation list --name <recovery-deploy> -g synthwatch-rg`** — per-resource
   `provisioningState`/`statusMessage`: does each `Microsoft.App/jobs` show a real update, a
   "no-change" skip, or did an earlier resource fail and halt the run? (Distinguishes no-op vs partial.)
2. **RG Activity Log at recovery time** — was there a `Microsoft.App/jobs write` during the
   `az deployment group create`, or did the only job write occur at the `--set-env-vars` timestamp?
   (Direct test of "the PUT never ran.")
3. **`az containerapp job show --query "properties.template.containers[0].env"` captured immediately
   AFTER the bicep apply and BEFORE `--set-env-vars`** — was the marker in the control-plane template
   or not? (Separates "ARM didn't write it" from "written but runtime didn't apply.")
4. **The archived what-if JSON from that deploy** (deploy.sh writes one to a temp file per run) — did
   what-if *predict* an env change on the jobs? If what-if showed no job-env delta, that is strong
   evidence for the idempotency-skip hypothesis.

**Caveat:** all four require live Azure/log access, outside this analysis-only scope. Until then this
stays a HYPOTHESIS; the repo + primary docs make the "idempotency no-op skipped the PUT" path the
best-supported, and definitively **refute** the revision-scope and deployment-stack candidates for
this deploy path.

## 6. A-vs-B re-measurement (T6.13) — the contradicted evidence, ground-truthed

Measured on `origin/main` (worktree `db/` verified `git diff --quiet origin/main -- db/` → identical).
Table-name extraction: `CREATE TABLE [IF NOT EXISTS] <name>` across the files, deduped.

| Metric | Ground truth |
|--------|--------------|
| `CREATE TABLE` in `db/schema.sql` | **31** (31 unique tables) |
| `CREATE TABLE` across `db/migrations/*.sql` | **30 statements** in 22 files → **26 unique tables** |
| Tables in schema.sql created by **NO** migration ("schema.sql-only") | **5** |
| Tables created by a migration but **absent** from schema.sql | **0** (schema.sql is the convergent superset) |
| A `0000` migration | **does not exist** — migrations start at `0001` |

**The 5 schema.sql-only tables (exact set):** `checks`, `incidents`, `runs`, `run_steps`,
`schema_migrations`. These are the foundational tables a fresh DB needs *before* any migration runs
(the check/run/step/incident core) plus the migration ledger itself. This matches the CLAUDE.md rule:
"the `checks` table is bootstrapped from `db/schema.sql` — no migration `CREATE`s it."

**Arithmetic checks out:** 26 (migration-created, all also in schema.sql) + 5 (schema.sql-only) = 31.

### Adjudicating the self-contradiction

- Banked claim **"5 tables schema.sql-only"** → **CORRECT.** (Exactly these 5.)
- Banked claim **"252-line 0000"** → **WRONG.** There is **no `0000` migration file at all.** (Whatever
  the 252-line artifact was, it is not a committed migration on main.)
- Later recon **"31 CREATE TABLE in schema.sql"** → **CORRECT.**
- Later recon **"0 CREATE TABLE in migrations"** → **WRONG.** 26 unique tables (30 `CREATE TABLE`
  statements) are created across the migrations.

**So both banks were half-right.** The reconciled ground truth for the A-vs-B decision: **schema.sql
is a convergent superset of the migration-created schema** — it holds all 26 migration-created tables
*plus* the 5 bootstrap-only core tables. Migrations do **not** re-create the core 5; a fresh env
therefore depends on `schema.sql` for `checks`/`runs`/`run_steps`/`incidents`/`schema_migrations`, and
on the 22 migrations for the other 26. The A-vs-B choice should rest on these numbers, not on either
banked figure.

**Falsifier (run):** "if migrations created 0 tables, `grep -c 'CREATE TABLE' db/migrations/*.sql`
would sum to 0." → It sums to 30 (26 unique). "If schema.sql-only ≠ 5, `comm -23` of the two
sorted name sets would differ." → It is exactly the 5 named above.
